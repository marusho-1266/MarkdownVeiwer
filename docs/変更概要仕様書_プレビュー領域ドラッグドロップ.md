# 変更概要仕様書：プレビュー領域へのドラッグ＆ドロップ読み込み

| 項目 | 内容 |

| ドキュメントID | MDV-DD-002 |
| 作成日 | 2026-04-22 |
| 最終更新 | 2026-04-22（§4.3 `dragleave` / `relatedTarget === null` の扱いを明示） |
| 対象機能 | 右側プレビュー表示領域（`#preview-wrapper`）へのローカル Markdown／テキストファイルのドラッグ＆ドロップによる読み込み |
| 対象ファイル | `index.html` |
| 関連仕様 | MDV-DD-001（左側入力エリアのドラッグ＆ドロップ）。本変更はその**読み込み処理の再利用**と、プレビュー側の**ドロップ無効化の撤廃**が中心である。 |

---

## 1. 変更の目的

- ユーザーがエクスプローラー等からファイルをドラッグしたとき、**左の入力エリアだけでなく、右側のプレビュー表示領域上にドロップしても**、同一の読み込み結果（テキストエリアへの反映とプレビュー更新）が得られるようにする。
- エディタを広く使わずプレビューに視線が乗っている場面や、パネルレイアウト上「右側に落としやすい」操作習慣に合わせ、**ドロップ先の選択肢を増やして利用性を向上**させる。

---

## 2. 対象範囲

| 区分 | 対象 |
|------|------|
| 変更対象 | 本システムのフロントエンド（単一 HTML アプリケーション） |
| 変更ファイル | `index.html`（`#preview-wrapper` 向けのドラッグ＆ドロップ用イベントハンドラの修正、および必要に応じた CSS の追加） |
| 非対象 | サーバー側処理、バックエンド API、`loadDroppedMarkdownFile` の文字コード・拡張子判定ロジックの**仕様変更**（MDV-DD-001 と同一のまま再利用する）、印刷・テーマ・Mermaid・行番号等の**別機能の仕様変更** |

---

## 3. 変更内容の概要

- **現状**  
  `#preview-wrapper` では `dragover` で `dropEffect = 'none'` とし、`drop` ではファイルを処理しない実装となっており、**プレビュー上へのドロップでは読み込みが行われない**。
- **変更後**  
  `#preview-wrapper` 上でも、左側 `#input-area` と**同等の**ドロップ受付を行う。具体的には、`DataTransfer.files` の先頭ファイルに対し、既存の **`loadDroppedMarkdownFile`** を呼び出す。
- **ブラウザ既定挙動の抑止**  
  `#preview-wrapper` 配下では、`dragover` および `drop` で**ファイルドロップ時の既定のナビゲーション**（別タブで開く等）を抑止し、左側と同様のパターンで統一する。子要素（リンク、スクロール領域等）上へのドロップでも、バブリングにより同じ方針が適用されること。
- **既存機能との関係**  
  ドロップ後の流れは MDV-DD-001 と同様とする。`#markdown-input` の `value` 更新と `input` イベント発火により、プレビュー・行番号・Mermaid 等は既存どおり更新される。

## 4. 機能仕様

### 4.1 ドロップ受付領域

- **対象要素**  
  右側のプレビューラッパー `#preview-wrapper` 全体をドロップ受付領域とする（プレビュー内に生成される子要素上へのドロップも、バブリングにより同様に処理されることを想定する）。
- **レイアウト・ヒット領域**  
  ラッパーおよび想定どおりの子要素上で、**一貫して**ドロップを受け付けること。実装では `pointer-events` 等によりドロップ先が無効化されないよう、左側と同程度の操作感が得られること。プレビュー本文がスクロールする場合も、スクロール後の表示領域・余白を含め、ユーザーが `#preview-wrapper` 系の領域と認識できる範囲でイベントが届く前提とする（詳細は第8節の受け入れ観点）。
- **左側との関係**  
  `#input-area` に対する既存のドロップ受付は**維持**する。左・右のいずれにドロップしても同じ読み込み処理が走る。

### 4.2 対象ファイル・文字コード・エラー扱い

- **拡張子・複数ファイル・文字コード（BOM 優先・Shift_JIS 読み直し）・非対応ファイル・読み取り失敗・既存編集内容の上書き**  
  いずれも **MDV-DD-001 の 4.2 / 4.3 / 4.4 / 4.5 / 4.6** に従う。実装では `loadDroppedMarkdownFile` を改変せず呼び出す前提とする。

### 4.3 動作フロー（プレビュー領域）

1. ユーザーがファイルを `#preview-wrapper` 上にドラッグする（`dragover`）。
2. ブラウザの既定挙動（例: 別タブでファイルを開く）を防ぐため `preventDefault` 等を行い、**ドロップ可能**であることを示す（`dropEffect = 'copy'`）。
3. **視覚的フィードバック（必須）**  
   ファイルが `#preview-wrapper` 上でドラッグオーバーされているあいだ、`dragover` のたびに左側 `#input-area` の `input-area-dragover` と**同等の見た目**となるフィードバックを出すこと。実装では `#preview-wrapper` にフィードバック用クラス **`preview-dragover`** を付与する。スタイルは `index.html` 内の `#input-area.input-area-dragover` と**同一の宣言パターン**（`outline`・`background-color` 等）に合わせ、`#preview-wrapper.preview-dragover` として定義する。
4. **`dragleave`**（`DragEvent.relatedTarget` を参照）でフィードバック用クラス **`preview-dragover`** を外す条件は、**MDV-DD-001 4.3** における左 **`#input-area`** の **`dragleave`** と同一とする。すなわち、**`#preview-wrapper.contains(event.relatedTarget)` が偽のときだけ**ハイライトを解除する（子要素間の移動では `relatedTarget` がラッパー内のノードになるためちらつかない）。**`relatedTarget === null`** のときは DOM の **`Node.contains(null)` が `false` を返す**ため上記条件は真となり、**ハイライトを解除する**（左 `#input-area` と同じガード式 `!area.contains(relatedTarget)` を用いる実装と一致する）。逆に、ラッパー内への移動だけでは解除しない。
5. **`drop`** では、読み込み処理の前にフィードバック用クラス **`preview-dragover`** を**必ず**削除する（非対応ファイル・`files` 空などで `loadDroppedMarkdownFile` を呼ばない場合も含む）。続けて `e.dataTransfer.files` の**先頭 1 ファイル**を対象とし、該当時は `loadDroppedMarkdownFile(files[0])` を呼ぶ。
6. 以降は MDV-DD-001 4.3 の手順 4〜6 と同じ（テキスト反映・`input` 発火・プレビュー更新）。

### 4.4 エディタ折りたたみ時との関係

- エディタペイン折りたたみ時の **左ガター／「エディタを表示」ボタン上のドロップ**（既存実装）は**変更しない**。プレビュー領域へのドロップは、折りたたみ状態に依らず**常に**受け付ける。

## 5. 非機能要件（概要）

| 項目 | 内容 |
|------|------|
| ブラウザ | HTML5 のドラッグ＆ドロップ API および File API をサポートするモダンブラウザ。MDV-DD-001 と同等。 |
| セキュリティ | ユーザーが明示的にドロップしたファイルのみを読み取る。サーバーへファイルを送信しない。 |
| パフォーマンス | 読み取りは既存の非同期 `FileReader` パターンのままとする。 |
| アクセシビリティ | プレビュー領域をドロップゾーンとして扱う場合、`aria-label` 等の付与を検討する（MDV-DD-001 第5節と同趣旨）。**本変更により、キーボードのみの新規操作フローを追加しない**（ドラッグ＆ドロップ以外の入力手段の要望は MDV-DD-001 の方針に従う）。 |

## 6. 影響範囲

- **既存機能**  
  テキスト入力・左側ドロップ・印刷・テーマ・Mermaid・リサイザ等は、プレビュー側でドロップを**許可する**変更以外の仕様変更を行わない。
- **デプロイ・ビルド**  
  単一 HTML の修正のみであり、ビルド・デプロイ手順の変更は不要。
- **ドキュメント**  
  `README.md` の使い方に、「プレビュー表示領域にドロップしてもファイルを読み込める」旨の一文追加を行う。**リリースに本変更を含める場合は当該 README 更新を必須**とし、別チケットに切り出す場合は必ず追跡すること。

---

## 7. 制約・注意事項

- プレビュー内のリンククリックやスクロールと競合しないよう、`drop` / `dragover`（必要に応じ `dragleave`）に処理を集約し、不要なグローバルリスナの追加は避ける。
- **複数ファイル**を同時にドロップした場合は、**先頭 1 件のみ**を対象とし、それ以外は MDV-DD-001 と同様に扱う（ユーザーへの追加通知を新設しない前提。詳細は付録 A および MDV-DD-001）。
- MDV-DD-001 7 の制約（ローカル／HTTPS、`file://` の扱い、ユーザー操作に起因する読み込みに限る等）を踏襲する。

---

## 8. 受け入れ条件（概要）

- `.md` / `.markdown` / `.txt`（既存の許可リストどおり）を `#preview-wrapper` 上にドロップすると、左側テキストエリアに内容が表示され、右側プレビュー・行番号・Mermaid が既存どおり更新されること。
- **レイアウト観点**：プレビュー本文を**スクロールした状態**、およびラッパー内の**余白**へドロップした場合でも、上記と同様に読み込みが行われること（操作感が著しく欠ける場合は不具合として扱う）。
- **複数ファイル同時ドロップ**時は**先頭 1 ファイルのみ**が処理され、2 件目以降の扱いが MDV-DD-001 と矛盾しないこと。
- UTF-8 および Shift_JIS の扱いが MDV-DD-001 4.6 と矛盾しないこと。
- 非対応ファイル・読み取り失敗時に、既存と同様の案内が行われること。
- `#input-area` へのドロップおよびエディタ折りたたみ時のガター／ボタンへのドロップの既存動作に退行がないこと。

### 8.1 テスト・回帰

- 上記は**手動確認**を主とする。E2E や自動テストの有無はプロジェクト方針に従い、本機能を含む**回帰シナリオ**（手動チェックリストまたは自動化）をチケットまたはテスト計画で管理すること。
